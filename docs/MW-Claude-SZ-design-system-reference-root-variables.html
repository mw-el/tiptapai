Farb- und Schriftkontrast

Jede Farbkombination im System erfüllt mindestens WCAG AA (Kontrastverhältnis ≥ 4.5:1 für normalen Text). Die nachfolgenden Regeln sind verbindlich.

Diese Kombination ist im HTML verbindlich hinterlegt

Die von dir festgelegte Zuordnung steht in dieser Referenzdatei direkt als Root-Variablen und als Textregel im CSS: dunkler Text ist immer SZ Deep Sienna auf hellen Flächen, und heller Text ist immer SZ Soft Paper auf dunklen Flächen.

}

Verbindliche Grundlagen

Designprinzipien

Diese Prinzipien gelten für alle Oberflächen, die auf diesem System basieren. Sie sind nicht optional.

palette

Farbe ist funktional, nicht dekorativ

Jede Farbe hat eine klar definierte Rolle. Orange ist ausschliesslich dem primären CTA vorbehalten. Lime signalisiert positive Zustände. Rot steht für Fehler, Gelb für Warnungen. Keine dieser Farben wird aus ästhetischen Gründen eingesetzt.

text_fields

Typografie folgt einer klaren Hierarchie

Fira Sans führt als Display-Schrift alle Überschriften. Plus Jakarta Sans trägt den gesamten Fliesstext, Labels und UI-Beschriftungen. Fira Code wird ausschliesslich für Code und technische Werte verwendet. Diese Rollen sind nicht austauschbar.

space_bar

Abstände kommen aus dem Token-System

Alle Innen- und Aussenabstände werden ausschliesslich über die definierten Spacing-Tokens gesetzt. Keine manuell gesetzten Pixel-Werte. Der Basis-Abstand ist 4px, die Skala läuft in Stufen von --space-4 bis --space-16.

layers

Elevation durch Schatten, nicht durch Farbe

Tiefe und Hierarchie entstehen ausschliesslich durch Schatten (--shadow-1, --shadow-2). Hintergrundfarben dürfen nicht zur Tiefendarstellung eingesetzt werden. Die Akzentlinie an hervorgehobenen Kacheln ist eine Ausnahme mit genau einer definierten Anwendung.

grid_view

Icons ausschliesslich aus Material Symbols

Alle Icons kommen aus Google Material Symbols Rounded. Keine SVGs aus anderen Sets, keine Emoji-Substitution. Gewicht und Fill-Grad werden per font-variation-settings gesteuert. Semantische Statusicons erhalten die jeweilige Systemfarbe.

brightness_6

Light und Dark sind gleichwertig

Beide Modi sind vollständig ausgearbeitet und teilen dieselbe Token-Basis. Kein Layout darf nur für einen Modus entworfen werden. Dark-Oberflächen nutzen ausschliesslich sz-ink-* Texttokens; Light-Oberflächen ausschliesslich sz-text-*.

rounded_corner

Radien sind kontextabhängig

Panels und grosse Container verwenden --radius-xl. Standard-Kacheln und Karten verwenden --radius-lg. Buttons und Eingabefelder verwenden --radius-md. Kleinere Elemente wie Tags nutzen 999px für vollständig runde Formen.

accessibility

Kontrast und Grösse sind Pflicht

Alle interaktiven Elemente haben eine Mindest-Touch-Target-Grösse von 44×44px. Texte auf farbigen Flächen müssen WCAG AA erfüllen. Eingabefelder haben immer einen sichtbaren Fokus-Ring in --brightgreen mit 2px Outline und 2px Offset.

Kachelbeispiele

Drei klar unterscheidbare Kachelvarianten: ruhig zurückhaltend, sanft angehoben und hervorgehoben mit feiner Akzentlinie am äußeren Rand.

Ohne Schatten

Ruhige Basiskachel

Diese Variante bleibt bewusst zurückhaltend und trennt sich nur über Fläche, Radius und eine feine Kontur vom Hintergrund.

Standard

Kachel mit Standardschatten

Hier bleibt der bestehende weiche Schattencharakter erhalten. Die Karte wirkt angehoben, ohne lauter oder dekorativer zu werden.

Hervorgehoben

Vorgehobene Akzentkachel

Diese Variante sitzt sichtbar weiter vorne und bekommt zusätzlich nur eine einzelne Haarlinie in der Akzentfarbe direkt am äußeren Rand.

Quote im Fließtext

Ein Beispiel dafür, wie ein Zitat innerhalb eines normalen Textflusses hervorgehoben werden kann, ohne typografisch aus dem System herauszufallen.

Guter Fließtext braucht eine klare Grundruhe. Ein Zitat sollte deshalb nicht einfach nur größer oder kursiv werden, sondern einen eigenen Takt im Layout bekommen.

Ein gutes Zitat fällt nicht aus dem Raster. Es verändert nur kurz den Tonfall und kehrt dann sauber in den Lesefluss zurück.

Beispielzitat · typografische Auszeichnung im Lesefluss

Die Einbindung wirkt am besten, wenn Einzug, Kontrast, Zeilenlänge und Weißraum präzise austariert sind und der Text davor sowie danach wieder ruhig weiterläuft.

Gestaltungsprinzip

Das Zitat nutzt dieselbe Grundschrift wie der Fließtext, bekommt aber mehr Größe, weniger Zeilenlänge und eine warme Akzentkante. So bleibt es verwandt mit dem System und wirkt dennoch besonders.

Plattform-Leitfaden

Web · Electron · PyWebView

Dieses Designsystem ist für drei Zielplattformen spezifiziert. Was nicht hier beschrieben ist, lässt sich analog ableiten.

language

Reines Web (Browser)

Die Baseline. Alle Tokens, Schriften und Komponenten funktionieren ohne Anpassungen.

  • Google Fonts via CDN — kein lokales Einbetten nötig
  • Material Symbols via CDN — variable Font, volle Unterstützung
  • CSS Custom Properties nativ verfügbar
  • Theme-Toggle via data-theme + localStorage
  • Kein Framework nötig — reines CSS und Vanilla JS genügen
  • Fokus-Ring outline funktioniert in allen modernen Browsern
desktop_windows

Electron

Chromium-basiert. Fast identisch mit Web, aber Fonts und Icons müssen für Offline-Betrieb lokal eingebettet sein.

  • Fonts lokal einbetten — kein CDN-Zugriff im Produktivbetrieb garantiert. Fira Sans, Plus Jakarta Sans und Fira Code als @font-face einbinden
  • Material Symbols lokal — WOFF2-Datei der Symbols Rounded Variable Font bündeln
  • Scrollbar-Styling mit ::-webkit-scrollbar möglich und empfohlen
  • Systemschriften als Fallback im Font-Stack definiert (system-ui)
  • IPC-Kommunikation über window.api — kein Fetch zu externen Ressourcen im Main-Process
  • CSP in BrowserWindow: font-src 'self' explizit erlauben
  • macOS: -webkit-app-region: drag für benutzerdefinierte Titelleiste nutzen
web_asset

PyWebView

WebKit auf macOS/Linux, Chromium (WebView2) auf Windows. Rendering-Unterschiede zwischen Plattformen sind einzuplanen.

Fonts

  • Ausschliesslich lokale Fonts — kein Google Fonts CDN
  • @font-face mit relativem Pfad zur Font-Datei
  • Variable Fonts (Fira Sans, Plus Jakarta Sans) werden auf WebKit und WebView2 unterstützt
  • Fira Code als Mono — WOFF2 mitliefern

Icons

  • Material Symbols WOFF2 lokal bündeln
  • font-variation-settings wird auf beiden Engines unterstützt
  • Icons im static/-Verzeichnis ablegen, das PyWebView als Root einbindet

Plattformdifferenzen

  • macOS WebKit: Schriften rendern leicht dünner — Gewicht 500 statt 400 für Body empfohlen
  • Windows WebView2: Subpixel-Rendering — Schriften wirken schärfer, Abstände stimmen gut überein
  • Linux: GTK-Rendering kann abweichen — mit Gewicht 400 testen
  • Kein backdrop-filter auf WebKit ohne Präfix
